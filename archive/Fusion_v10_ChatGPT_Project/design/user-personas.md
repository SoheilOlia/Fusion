# User Personas: Understanding Our Users

## Primary Persona: Margaret (72, Widowed)

### Context
- Lives alone, adult children visit weekly
- Has diabetes, mild arthritis, takes 4 medications daily
- Uses iPad for email, struggles with small phone buttons
- Values independence but worries about emergencies

### Needs
- **Primary**: Feel safe and independent
- **Secondary**: Stay connected with family and healthcare team
- **Hidden**: Maintain dignity while getting help

### Frustrations with Technology
- "Too many steps to do simple things"
- "Buttons are too small, text too tiny"
- "Apps keep changing - just when I learn them"

### Healthcare Challenges
- Forgets medication timing
- Anxious about emergency situations
- Wants to share health data with daughter
- Doctor visits feel rushed

## Secondary Persona: Robert (68, Married)

### Context
- Recent heart attack, now very health-conscious
- Wife helps with technology, both learning together
- Retired engineer, appreciates well-designed systems
- Skeptical of "dumbed down" interfaces

### Needs
- **Primary**: Comprehensive health monitoring
- **Secondary**: Data that doctors can actually use
- **Hidden**: Feeling useful and capable

### Design Requirements from Robert
- Don't hide advanced features - make them discoverable
- Provide detailed data when requested
- Respect intelligence while accommodating physical limitations

## Design Implications

### For Margaret: Simplicity First
- Large touch targets (minimum 44pt)
- High contrast, clear fonts
- Consistent navigation patterns
- Immediate feedback for all actions

### For Robert: Power Under the Hood
- Advanced features accessible but not prominent
- Data export capabilities
- Customizable dashboard
- Detailed health analytics

### Universal Needs
- Emergency access from any screen
- Voice alternatives for all inputs
- Offline functionality for core features
- Family sharing and oversight options
